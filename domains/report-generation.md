# Report Generation

For when the output is a document somebody will actually print, scroll, or attach to an email — not a notebook, not a README, not a slide deck. PDF documents are how technical work earns belief from non-technical readers, and they earn it by looking like the writer cared. The single biggest gain available in 2026 is using the HTML+headless-Chromium toolchain or Quarto/Typst to make Claude-built artefacts look engineered, not pasted.

## Pick the stack first

Three patterns dominate. Pick before writing:

| Stack | Best for | Strengths | Weaknesses |
| --- | --- | --- | --- |
| **HTML + CSS print + headless Chromium** (Puppeteer / Playwright / weasyprint) | Designed deliverables, branded client outputs, anything with rich layout | Full CSS, web fonts, easy figures, live-preview in browser | More setup; heavier toolchain |
| **Quarto** (Pandoc + LaTeX/Typst backend) | Long-form technical, papers, books, anything with maths and citations | Citations, cross-refs, code execution, one source → PDF/HTML/docx | LaTeX dependency for nicest output; slower iteration |
| **Typst** | Modern technical writing, theses, anything LaTeX-shaped but less painful | Fast, clean syntax, good defaults | Smaller ecosystem; fewer templates |

For a one-off product-launch one-pager: HTML+Chromium. For a 40-page modelling write-up with references and equations: Quarto with Typst backend. For a thesis: Typst direct.

## Phase 1 — Frame the document

Before writing any prose:

- **Who is the reader?** Technical peer, technical manager, executive, customer, regulator. Each wants a different opening.
- **What decision does this support?** Documents without a decision are journals. State the decision in one sentence at the top of the brief.
- **What is the single takeaway?** If the reader reads only the cover, what do they need to know?
- **What length is appropriate?** 2-page brief, 10-page summary, 40-page full document. Pick before you start; cut to fit, don't expand to fill.
- **What evidence is load-bearing?** Which figures, tables, citations must appear. The rest is supporting.

Skip this phase and the document will sprawl. The first symptom of a sprawling document is the writer adding an executive summary at the end.

## Phase 2 — Outline before prose

The structure for a technical document that earns belief:

1. **Cover** — title, subtitle, author, date, version, single hero figure or none
2. **Executive summary** — half-page max; the takeaway, the evidence in three sentences, the recommendation
3. **Context** — what problem, why now, what was asked
4. **Approach** — what was done, in enough detail that a peer could repeat it
5. **Results** — figures and tables carry the load; prose explains them, doesn't restate them
6. **Discussion / limitations** — what the numbers do and don't support; honest gaps
7. **Recommendation** — what the reader should now do
8. **Appendices** — derivations, supplementary plots, parameter tables

For a 2-page brief: collapse to cover + summary + 1-page evidence + recommendation. Same structure, compressed.

## Phase 3 — Build the toolchain

Project layout:

```
doc/
├── doc.md               # or .qmd / .typ — the source
├── refs.bib             # citations (if any)
├── figures/             # generated by code, not handcrafted
├── data/                # inputs (small, committed)
├── build/               # outputs (gitignored)
├── Makefile             # `make doc.pdf` reproduces everything
└── assets/              # logos, fonts, CSS
```

The build must be one command, deterministic, reproducible from a clean clone. If the figures aren't generated by code, the document rots the moment data changes.

### HTML+Chromium recipe

- Author in semantic HTML (`<h1>`, `<section>`, `<figure>`, `<table>`)
- CSS with `@media print` for A4/Letter page sizes, page breaks, headers/footers with page numbers, cover page
- Web fonts loaded locally for reproducibility
- Charts as SVG (matplotlib `savefig('fig.svg')`) so they stay sharp at any zoom
- Build: `playwright pdf doc.html doc.pdf` or `weasyprint doc.html doc.pdf`

### Quarto recipe

- `doc.qmd` with YAML header (title, author, date, format)
- Inline code blocks generate figures and tables at build time
- Citations via `@citekey`; `refs.bib` handles the rest
- Build: `quarto render doc.qmd --to pdf`

### Typst recipe

- `doc.typ` with `#set page(...)` and `#show heading: ...` for styling
- Functions for repeated elements (callouts, parameter tables)
- Build: `typst compile doc.typ`

## Phase 4 — Figures and tables earn their place

Each figure answers one question. Each table is a data structure the prose can't substitute for. Both get:

- A number and a caption
- Cited in the prose by number ("Figure 3 shows…")
- Axes labels with units, legend, source line
- Consistent style across the document (same fonts, same colour palette, same DPI)
- For tables: aligned numbers, units in the header row, no decoration that doesn't carry information

If a figure or table isn't cited in the prose, delete it.

## Phase 5 — Citations and provenance

For any number that isn't your own computation:

- Citation in `refs.bib` or equivalent
- Footnote or inline cite at first use
- A `data-sources.md` if multiple datasets are pulled in

The reader's first defence against a wrong number is "where did this come from?" — make the answer trivially findable.

## Phase 6 — The cover page rule

Designed documents earn belief in the first two seconds. The cover page does that work:

- Real typography (system font is fine; Comic Sans is not)
- One hero figure or none — never a stock photo, never a decorative graphic that doesn't carry meaning
- Title that says what the document is about, not "Report" or "Analysis"
- Author, date, version. If draft, mark it draft loudly. If confidential, mark it.

Cover page failure is the single most common reason a strong document is treated as weak.

## Phase 7 — The skim test

Before delivery, skim the PDF at one page per second. Should be possible to extract:

- The takeaway (from the cover or summary)
- The shape of the evidence (from the figures)
- The recommendation (from the closing section)

If skimming doesn't surface those, the document is structured for the writer, not the reader. Restructure.

## Voice and prose

- Lead with the answer; justify after.
- Specific verbs. Numbers with units. No "very", "quite", "significant" without a number attached.
- One idea per paragraph. Paragraphs of fewer than two sentences usually mean a list was the wrong choice; paragraphs over six sentences usually mean the paragraph is doing two jobs.
- No filler ("In conclusion…", "It is important to note that…"). Cut.

## When NOT to apply

- The deliverable is a slide deck → use the slide-deck skill (when authored), not this one
- The deliverable is a paper for a journal → use `academic-writing.md`
- The deliverable is internal-team documentation → README + checked-in markdown is faster and better-version-controlled
- The reader will open it once and never read past page 2 → write a 2-page brief, not a 30-page document

## Anti-patterns

- Writing prose first, structure second
- Stock photos, decorative banners, gradient backgrounds — visual noise reads as desperation
- Figures pasted from screenshots instead of regenerated from code
- Citations only in the appendix — readers want them inline at first use
- "Executive summary" that's actually an introduction
- One-source PDF with no reproducible build — six months later, nobody can update a number without rebuilding from scratch
